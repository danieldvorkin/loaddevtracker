# Load Dev Tracker

A minimal load development tracker built with React, TypeScript, Firebase, and XState.

This document explains the app's architecture and the key design choices, especially around state management with XState and Firestore synchronization.

## Quick Start

- Install dependencies: `npm install`
- Run local Firebase emulators (recommended during development):

```bash
firebase emulators:start --only firestore,auth
```

- Start the dev server: `npm run dev`

## Project Structure (high level)

- **src/machines/loadMachine.ts**: Core state machine that models load creation and entry workflows.
- **src/lib/firebase.ts**: Firebase initialization and emulator connection logic.
- **src/pages/loads/**: UI pages for creating and viewing loads.
- **src/components/**: Reusable components such as `LoadChart` and `Toast`.

See the files in the repository for detail, for example: [src/machines/loadMachine.ts](src/machines/loadMachine.ts) and [src/lib/firebase.ts](src/lib/firebase.ts).

## Main Design Goals

- Predictable state management: use `xstate` to model the load lifecycle and explicit transitions.
- Local-first UX: UI updates (add/update/remove entry) should be immediate (optimistic) while persistence happens in the background.
- Simple offline/dev workflow: support Firebase emulators during development and anonymous auth for background writes.

## State Machine Overview (`loadMachine`)

The `loadMachine` (source: [src/machines/loadMachine.ts](src/machines/loadMachine.ts)) encapsulates the lifecycle of a single load document and its entries. Key parts below.

- Context
  - `loadData`: nullable object representing the current load document: `{ id: string, name: string, description?, entries: Entry[] }`.
  - `error`: nullable error object `{ message, stack?, raw? }`.

- Events
  - `CREATE` — start creating a new load (from UI). Payload: `{ name, description? }`.
  - `CREATED` — used by UI after Firestore document is created to set returned data (id, entries, etc.).
  - `ERROR` — used to set error state.
  - `ADD_ENTRY` — add an entry locally. Payload: `entry` (omits `id`).
  - `UPDATE_ENTRY` — update a full `Entry` (with `id`).
  - `REMOVE_ENTRY` — remove an entry by `entryId`.
  - `RESET` — clear the machine and go back to `idle`.
  - `RETRY` — retry creating after failure.

- States
  - `idle`: waiting to create a load.
  - `creating`: intermediate state — UI performs Firestore `create` and then sends `CREATED` or `ERROR`.
  - `ready`: load exists locally and is editable. Local entry mutations are applied here.
  - `failure`: creation failed; can `RETRY` or `RESET`.

### Key Actions (what they do)

- `setLoadData`:
  - Assigns `loadData` with a normalized object (preserves existing context where appropriate).
  - Clears `error`.

- `setError`:
  - Assigns an error object to `context.error`, with safe fallbacks (ensures message and optional stack/raw fields).

- `clearLoadData`:
  - Resets `loadData` and `error` to `null`.

- `addEntryLocally` / `updateEntryLocally` / `removeEntryLocally`:
  - Perform immediate, local changes to `loadData.entries` using `assign` so UI updates instantly.
  - `addEntryLocally` constructs a new `Entry` with `id: Date.now()` to create a client-side unique id.

- `syncEntries`:
  - Background persistence action invoked after local mutations.
  - Ensures anonymous auth by calling `signInAnonymously` if needed, then calls `updateDoc` to update the document's `entries` array in Firestore.
  - Errors are caught and logged but do not roll back local state.

Design rationale: these actions implement an optimistic, local-first UX: update the UI synchronously, then persist in background. This reduces perceived latency and keeps the UX snappy.

## Firebase integration (`src/lib/firebase.ts`)

- App initialized via `initializeApp` with `VITE_FIREBASE_*` env vars.
- Connects to local emulators automatically when `VITE_USE_FIREBASE_EMULATOR=true` or when `window.location.hostname` is `localhost`/`127.0.0.1`.
- Exports `auth` and `db` for use by `loadMachine` and other modules.

Developer notes:

- Ensure required env vars exist; `src/lib/firebase.ts` throws early if critical keys are missing.
- Use the Firebase emulator suite during development to avoid touching production data.

## Data Model Notes

- `Load` (document):
  - `id`: Firestore document id (string).
  - `name`, `description`.
  - `entries`: array of `Entry` objects.

- `Entry`:
  - In `loadMachine`, `Entry.id` is a `number` (generated by `Date.now()` for local IDs). This is intentionally different from Firestore document id (string) because entries are stored as an array inside a load document rather than as sub-collection documents.
  - If you later migrate entries to named documents (subcollection), consider using string UUIDs for consistency.

## Error Handling & Resilience

- `syncEntries` logs failures but does not raise user-facing errors by default. This keeps local state stable even if Firestore is temporarily unreachable.
- `setError` captures and normalizes errors so UIs can present consistent error messages.

## Trade-offs & Alternatives

- Current approach uses XState actions (and an async IIFE inside `syncEntries`) for persistence. This keeps the machine simple but mixes side-effects inside actions.

Alternatives to consider:

- Move persistence to invoked services (spawned actors) so you can model in-flight sync, retries, and cancellation explicitly in the statechart.
- Add a write queue / retry strategy for `syncEntries` with exponential backoff and per-change versioning to reduce overwrite risk.
- Use server-side timestamps or UUIDs for entry ids if you need cross-client deterministic ids.

## Suggested Improvements

- Show sync status in the UI (pending / synced / failed) per entry or per load.
- Implement idempotent updates (use a `version` or last-modified field) to reduce concurrent overwrite issues.
- Add unit tests for `loadMachine` using `@xstate/test` or `@xstate/fsm` to validate transitions and assign behaviors.
